# PvZ Win32 Memory Modifier - 逆向分析报告

## 概述

本文档记录了植物大战僵尸中文版内存修改工具的开发过程，包括使用 Cheat Engine 和 x64dbg 进行逆向分析的完整方法论。

## 环境准备

| 工具 | 版本 | 用途 |
|------|------|------|
| Cheat Engine | 7.5 | 内存扫描、指针链分析、代码注入测试 |
| x64dbg | 2024+ | 动态调试、汇编级代码分析 |
| Process Hacker | 2.x | 进程查看、句柄监控 |
| IDA Free | 8.x | 静态反汇编（辅助分析） |

## 一、阳光值偏移定位

### 1.1 初始扫描

```
步骤1: 启动游戏，进入关卡
步骤2: 打开 Cheat Engine，附加到 PlantsVsZombies.exe
步骤3: 记录当前阳光值（如 150）
步骤4: 在 CE 中搜索精确值 150（4字节）
步骤5: 种植植物消耗阳光，搜索新值
步骤6: 重复步骤5直到仅剩数个结果
```

### 1.2 指针链分析

找到动态地址后，使用 CE 的 "找出是什么访问了这个地址" 功能：

```
结果分析:
  阳光地址 = [esi + 0x5560]
  esi = [0x6A9EC0 + 0x768]

最终公式: [[0x6A9EC0 + 0x768] + 0x5560]
```

关键发现：`0x6A9EC0` 是一个**静态地址**（位于 `.data` 段），不受 ASLR 影响。

### 1.3 静态地址验证

使用 x64dbg 验证：
1. 在 `0x6A9EC0` 下硬件断点
2. 观察每次访问该地址的代码上下文
3. 确认该地址始终指向游戏对象基址

## 二、金币值偏移定位

方法与阳光相同，最终确定指针链：

```
金币地址 = [[0x6A9EC0 + 0x82C] + 0x28]
```

地址链验证（Python 伪代码）：
```python
base = read_u32(0x6A9EC0)
ptr1 = read_u32(base + 0x82C)
gold_addr = ptr1 + 0x28
gold_value = read_u32(gold_addr)
```

## 三、植物冷却 CD 代码定位

### 3.1 功能分析

植物冷却机制通过代码层面的条件跳转实现。目标是找到控制冷却计时的 JLE/JNE 指令。

### 3.2 CE 代码注入法

```
步骤1: 在游戏中观察，种植后开始冷却计时
步骤2: CE → 内存浏览器 → 搜索 "查找写入该地址的代码"
步骤3: 观察冷却计时器的递减模式
步骤4: 定位到递减后的比较/跳转指令
```

### 3.3 关键定位

**植物无冷却** `0x004875E6`:
```asm
004875E6 | 7E 14  | jle plantsvszombies.4875FC   ; 如果冷却结束则跳转
004875E8 | ...    | (冷却中执行的代码)
```

修改方案：将 `jle` 改为 `nop; nop` (90 90)，使冷却检查失效。

**大嘴花吞噬无冷却** `0x004616E5`:
```asm
004616E5 | 75 5F  | jne PlantsVsZombies.exe+61746
```

修改方案：同上，NOP 注入。

### 3.4 x64dbg 动态验证

在 x64dbg 中对此地址设置断点：
1. 正常游戏时触发断点（种植植物后）
2. 修改 `7E 14` → `90 90`
3. 确认冷却计时消失，验证功能正确

## 四、植物无敌代码定位

### 4.1 搜索伤害递减

植物每帧扣血通过 `add [esi+40], -04` 实现：
```asm
00530040 | 83 46 40 FC  | add dword ptr [esi+40], -04   ; 血量 -= 4
```

修改 `FC (-04)` → `00` 即可实现扣血无效化。

### 4.2 定位方法

使用 Cheat Engine 搜索植物血量变化模式：
1. 定位植物血量地址
2. "找出是什么写入"
3. 定位到 `add [esi+40], -04` 指令
4. 确认 `0x00530043` 处的 `FC` 字节

## 五、种植函数与 Shellcode 注入

### 5.1 函数发现

通过 CE 的 "找出是什么访问了地址" 在植物对象地址上设置断点：
```
创建植物 → 写入植物结构体 → 追溯到 SpawnPlant 函数
```

### 5.2 函数签名

```cpp
// 0x0040D130
void SpawnPlant(int slot, int plantType, int x, int y, void* gameObject);
```

- `slot`: -1 = 自动选择卡槽
- `plantType`: 2 = 樱桃炸弹
- `x`: 0~7 (列)
- `y`: 0~4 (行)
- `gameObject`: `[[0x6A9EC0] + 0x768]`

### 5.3 Shellcode 设计

由于 `SpawnPlant` 是游戏内部函数，需通过远程线程在目标进程空间内调用：

```asm
pushad                  ; 保存寄存器状态
push -1                 ; slot = 自动
push 2                  ; type = 樱桃炸弹
mov  eax, <y>           ; 动态填写 y 坐标
push <x>                ; 动态填写 x 坐标
mov  ebx, [0x6A9EC0]    ; 游戏基址
mov  ebx, [ebx+0x768]   ; 游戏对象
push ebx                ; this 指针
mov  edx, 0x40D130      ; SpawnPlant 地址
call edx                ; 调用种植函数
popad                   ; 恢复寄存器
ret                     ; 线程返回
```

注入流程：
1. `VirtualAllocEx` 分配远程可执行内存
2. 动态修改坐标占位符 (x, y)
3. `WriteProcessMemory` 写入 Shellcode
4. `CreateRemoteThread` 执行
5. `WaitForSingleObject` 等待完成
6. `VirtualFreeEx` 释放内存

## 六、版本兼容性说明

### 6.1 地址来源

上述地址基于**植物大战僵尸中文版**特定版本确定。不同版本的游戏可能因重新编译导致地址偏移变化。

### 6.2 应对版本更新

当游戏更新后地址失效时：
1. 使用 `tools/find_offset.py` 扫描特征码
2. 使用 CE 重新追踪指针链
3. 更新 `src/offsets.h` 中的宏定义

### 6.3 特征码参考

```
植物CD:     xx xx xx ... 7E 14 xx xx ...   (jle short +0x14)
大嘴花CD:   xx xx xx ... 75 5F xx xx ...   (jne short +0x5F)
植物无敌:   ... 83 46 40 FC ...            (add [esi+40], -04)
基址引用:   ... C0 9E 6A 00 ...            (0x006A9EC0 的字节序)
```

## 七、安全与稳定性

### 7.1 代码修改风险

- NOP 注入和字节修改直接操作游戏代码段（`.text`）
- 游戏重启后代码段从磁盘重新加载，修改自动恢复
- 全图樱桃炸弹会瞬间创建 40 个远程线程，建议添加延时

### 7.2 权限要求

- 需要 `SeDebugPrivilege` 权限（以管理员身份运行）
- 需要 `PROCESS_ALL_ACCESS` 进程访问权限

### 7.3 反检测

单机游戏无反作弊机制，但杀毒软件可能对 `WriteProcessMemory` 和 `CreateRemoteThread` 行为产生告警。
